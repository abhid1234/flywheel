flywheel · a research note

I gave a coding agent reinforcement learning on its own mistakes. Here is the honest curve.

An agent attempts tasks, gets a verifiable reward, and improves from its own graded trajectories — no weight updates, just lessons it writes for itself. It works. It also fails in exactly the ways it should, which is the more interesting half.

weekend research build · run live on Daytona sandboxes with Codex · RLagentshonest negatives

The pitch for self-improving agents is seductive and almost always oversold. An agent notices what it keeps getting wrong, fixes it, and gets a little better each cycle — a flywheel. The failure mode of every demo like this is the same: the agent grades its own homework, the numbers go up, and nobody measured anything real. So I built the version that refuses to do that, ran it on live infrastructure, and I am going to show you the parts that didn't work, because they are the parts that make the parts that did work believable.

The setup is reinforcement learning, at the context layer. The loop is the RL loop exactly: the agent attempts a task (a rollout), earns a verifiable reward — real code, executed against hidden tests it never sees — and its policy improves from its own trajectories. The one difference from textbook RL is where the update lands. There are no gradients and no GPUs. The "policy" is a set of durable lessons the agent carries in its context, and each round it distills a new one from its own clustered failures. Experience, replayed as instruction.

The model writes the fix. It never writes the success criterion.

Everything hangs on that rule. The reward comes from hidden tests, fixed in advance and out of the model's reach; improvement is only ever measured on a sealed held-out set the agent never learns from. The model can propose any lesson it likes. It cannot move the goalposts. What follows is what happened, in the order it happened — including two dead ends that turned out to be the most informative results of the week.

00the null result that had to come first

A strong model has no room to learn on easy tasks

The first live run used simple code-generation tasks — sum a list, handle an empty input. Codex scored 100% at baseline. There were no failures to harvest, so the loop distilled no lessons and converged immediately. That is not a disappointing run; it is the system's conscience. It refused to manufacture a learning curve on tasks the model already passes. Any framework that shows improvement here is measuring its own imagination — the exact thing this project exists to avoid.

01finding the frontier

Real learning lives where the model actually fails

To create headroom I wrote conventions tasks: the spec is deliberately underspecified on one point, and a hidden test enforces a convention the model has no way to infer — an empty input should return None, not raise; "unique values" means sorted, not input order; ranges are inclusive of both ends; dates are zero-padded ISO. Cold, a competent model guesses, and often guesses wrong. This is precisely how a real agent meets a codebase's implicit rules: by getting them wrong first.

On these, Codex's baseline dropped to 27% — and stayed at 27% in every single run afterward. A reliable floor is as valuable as a reliable ceiling: it is something to measure improvement from.

02the gate fires against a true positive

The loop refused to credit a real gain it couldn't resolve

The first powered attempt produced a genuine +17.9-point improvement from its first lesson — which landed just under the measurement's noise band and was rejected. The effect was real. The measurement was underpowered. And the system took the loss rather than credit a number it could not distinguish from run-to-run noise.

I want to dwell on this, because it is the whole ethic in one event. A system eager to impress keeps that +17.9. A system you can trust throws it away and tells you it needs more evidence. The fix is not cleverer code — it is more measurement. More rollouts and more held-out tasks lower the noise floor. So I raised the rollout count, the floor dropped, and the real gains started clearing.

03the curve, and whether it survives being run again

It climbs — and it reproduces

With adequate statistical power, a live Codex agent improved on the sealed held-out set across four rounds, learning lessons it wrote itself from the clustered failing transcripts. One of them, verbatim: "Infer output ordering from the tests as part of the contract, instead of assuming 'unique' means preserving input order." That is an agent articulating a codebase convention it discovered by failing.

One run is an anecdote. Codex is non-deterministic, so I ran the same configuration four times — each an independent draw — and asked the only question that matters: does the climb reproduce, or did one run get lucky?

Every run climbed. The baseline was identical — 27% — all four times. Mean final 68%, mean gain +40 points. The mechanism, an agent improving from its own graded failures, reproduces.

04the honest limit

What reproduces is the climb, not every individual lesson

Here is the boundary, and it is the most important table in the piece. Across four runs, no single lesson was kept every time — but reliability tracks effect size exactly as it should:

lessonkeptavg gainreading
sorted-unique3 / 418.2pplarge cluster → mostly reliable
parse-none2 / 418.8pplarge cluster → reliable when it clusters
empty-none2 / 412.7ppmarginal — effect ≈ the noise band
round-2dp0 / 30.5pp≈ zero true effect — never credited

The powered run is the tell. Its two big-cluster lessons cleared with a two-to-threefold margin — parse-none at +28 points off 20 recorded failures, sorted-unique at +23 off 10 — and drove the highest final. The lessons that reproduce are the ones backed by large, recurring failure clusters with an effect well above the floor. The flaky ones have a true effect sitting at the floor. The one that never cleared has essentially no effect at all — Codex handles that case cold.

So the claim I can defend is precise: an agent learning from its own recurring failures reproducibly improves. The claim I cannot yet defend — that any given lesson reliably helps — is one I should not be able to make, because some of those "lessons" target effects indistinguishable from noise.

05a sharper signal, for free

Which step in a failed run actually caused it

A single pass/fail at the end of a multi-step trajectory blames every action equally — including the good ones. Borrowing an idea from a Daytona talk at Stanford: snapshot the sandbox at each step, fork many continuations from each, and watch for the step where the continuations collapse from "some recover" to "all doomed." That step is where the mistake lives, and where the lesson belongs.

Forking turned out to be gated on my Daytona tier — so I built the fork-free equivalent, replaying each trajectory prefix instead. On a recorded five-step run that poisons its own state at step two, the continuation pass rate held at 100% through step two and dropped to 0% immediately after, locating the culprit exactly. It upgrades the loop's signal from "the task failed" to "the decision at step two sank it" — a far more precise thing to learn from.

06the bottom line

The difference between a demo and a result

I set out to see whether an agent could do reinforcement learning on its own trajectories at the context layer, and to measure it without fooling myself. What I have: a stable, reproducible baseline; a gate that refused a real gain it couldn't resolve; a curve that climbs across four independent runs; a clear, quantified boundary on what the measurement can and cannot attribute; and a step-level signal for where the real mistakes are.

The negative space is the trustworthy part. No headroom on easy tasks. A true positive rejected. Individual lessons that don't reproduce. A capability gated on the infrastructure tier, reported rather than hidden. None of it was rounded up. That is the whole point — and it is the only version of "self-improving agent" I would ask you to believe.
