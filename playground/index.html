the one rule everything is built around
The model writes the fix. It never writes the success criterion.

If a model can author its own passing test, a self-improvement loop optimizes for the model's imagination instead of reality. So the success criterion is derived deterministically from the observed failure — a witness: the exact command, in the exact directory, that failed in production.

✕ the imagination trap

Model proposes a fix and writes the test that grades it. The test passes. Nothing real was verified — the loop just learned to satisfy itself.

graded by: the model

✓ the flywheel rule

The criterion comes from a recorded failure the model can't touch. A fix is "verified" only when replaying that witness goes red → green.

graded by: reality

the finding that reframes everything

      
Witness replay only validates environmental fixes.

      
Replaying the failed command is a perfect oracle for one class of problem — and useless for another. This is the honest core of the project, and it's why "my agent self-improves" is usually an overclaim.

    

    

      

        
environmental the missing package

        
A script fails with ModuleNotFoundError. The fix — pip install pyyaml — changes the environment. Replay the exact command: it goes red → green. Trustworthy at n=1.

◑ inconclusive — needs the statistical arm, not a replay

      

    

    
The catch, stated plainly: the fixes replay can verify cheaply are the ones that matter least. Behavioural fixes — the failures that actually cluster — need a statistical arm run over many trials. Which leads to the one result this whole project exists to report honestly.

  







  

    

      
the pre-registered result · KC-6

      
The mechanism works. That it moves the outcome is unproven.

      
To prove a behavioural fix helps, you run the agent on many gold-labeled tasks, before and after, and beat the run-to-run noise. The math says n ≥ 60 for a 5-point effect. This is a wall you can't code past — only accumulate past.

015304560 · unlocks the statistical arm

        
The agent factory on the Mac mini emits one episode→outcome tuple per merge decision, with a structured exit-code verdict. Reaching 60 is a multi-day wait — which is itself the finding: if you want to learn from outcomes, you must record the episode→outcome link at decision time. Bolting it on afterward doesn't work.

Cross-checked against GitHub ground truth, it recovered 3 of 4 documented failures — and correctly refused to guess the 4th (a semantic defect invisible from exit codes).

        
✓ M2 · the closed loop is proven
A recorded witness, replayed unchanged after the real cause was fixed, went red → green. A note that couldn't fix a missing package was reported helped:false.

        
◑ M3 · the statistical arm is blocked, honestly
Not a bug — the pre-registered kill-criterion. Anyone claiming their self-improvement loop "works" without outcome-level stats on a held-out set is showing you the mechanism and calling it the outcome.
